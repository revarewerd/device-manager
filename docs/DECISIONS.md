## ADR-001: Static Partition ключ для device-commands

**Дата:** 2026-02-15  
**Статус:** Принято

**Контекст:**
Команда на трекер должна дойти до конкретного инстанса Connection Manager, где устройство подключено по TCP.

**Решение:**
Использовать `instanceId` Connection Manager в качестве ключа партиции Kafka. DM читает `instanceId` из Redis (`device:{imei}`) и публикует команду с этим ключом.

**Последствия:**
- (+) Команда гарантированно попадает в правильный инстанс CM
- (+) Не нужен broadcast по всем инстансам
- (-) Если CM перезапущен и устройство на другом инстансе — команда уйдёт в wrong partition
- (mitigation) CM при переподключении обновляет `instanceId` в Redis

## ADR-002: Redis HASH как shared state между DM и CM

**Дата:** 2026-02-10  
**Статус:** Принято

**Контекст:**
Connection Manager должен знать контекст устройства (orgId, speedLimit, hasGeozones) для обогащения GPS-точек. DM знает эти данные, CM — нет.

**Решение:**
Используем Redis HASH `device:{imei}` как shared state. DM записывает CONTEXT-поля при CRUD, CM записывает POSITION и CONNECTION поля при обработке GPS-пакетов.

**Последствия:**
- (+) Минимальная задержка (<1ms) при чтении контекста
- (+) Нет прямой зависимости CM → DM API
- (-) Eventual consistency — после обновления в DM до синхронизации Redis есть окно
- (-) Нужна ежедневная синхронизация для исправления drift

## ADR-003: Pending commands в Redis Sorted Set

**Дата:** 2026-02-20  
**Статус:** Принято

**Контекст:**
Когда трекер оффлайн, команду нельзя отправить. Нужно хранить очередь и отправить при подключении.

**Решение:**
`ZADD pending_commands:{imei}` с `score = timestamp`. При подключении CM читает `ZRANGEBYSCORE` и отправляет все команды по порядку. TTL = 24 часа.

**Последствия:**
- (+) Порядок команд сохраняется за счёт score
- (+) Автоматическое удаление устаревших команд через TTL
- (-) Потеря команд при Redis failover (acceptable для MVP)

## ADR-004: Ежедневная синхронизация Redis ↔ PostgreSQL

**Дата:** 2026-03-01  
**Статус:** Принято

**Контекст:**
Redis и PostgreSQL могут разойтись из-за race conditions, сбоев сети или перезапусков.

**Решение:**
Cron job в 03:00 UTC сканирует все `device:*` ключи, сравнивает с PostgreSQL, исправляет: orphaned → удалить из Redis, missing → записать из БД, drift → перезаписать из БД.

**Последствия:**
- (+) Гарантия eventual consistency
- (+) Логирование аномалий для отладки
- (-) Нагрузка на Redis SCAN и PostgreSQL SELECT — допустимо в ночное время

## ADR-005: UUID request_id для идемпотентности команд

**Дата:** 2026-02-25  
**Статус:** Принято

**Контекст:**
При retry или двойном нажатии кнопки команда может создаться дважды.

**Решение:**
Клиент генерирует `request_id` (UUID), DM проверяет уникальность через UNIQUE constraint. При дубле — возвращает существующую команду без повторного создания.

**Последствия:**
- (+) Безопасный retry без побочных эффектов
- (+) Аудит привязан к конкретному запросу
- (-) Клиент должен правильно генерировать UUID (если не передан — DM генерирует сам)
